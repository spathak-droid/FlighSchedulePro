↻ Data Flow — The Core Loop

The system runs autonomously. Schedulers just approve or decline.

1

Schedule Polling

Repeatable BullMQ job runs every 3 minutes. For each active operator, fetches schedule from FSP API for next 7 days. Hashes the schedule and compares to sync_state.lastScheduleHash. If changed → detects cancellations and gaps. Enqueues generate-suggestions job.

src/worker/jobs/poll-schedule.job.ts (24KB) src/core/scheduling/change-detector.ts
2

Suggestion Generation — 3 Parallel Pipelines

Reschedule: Find alternative slots for cancelled students via progressive window expansion (7 → 14 → 28 days)

Waitlist: Rank eligible students by configurable weights (wait time, progress, flight hours), evaluate constraints, pick top 5

Next-Lesson: For students with no upcoming booking but pending lessons — determine next curriculum lesson, find slots

Each pipeline uses: slot-finder → constraint-evaluator → waitlist-ranker → rationale-builder

src/worker/jobs/generate-suggestions.job.ts (38KB)
3

AI Enrichment

Async job calls Claude Haiku 4.5 (via OpenRouter). Generates natural-language rationale + risk assessment (low/medium/high). Stores in suggestions.rationale JSONB. GPT-4.1-nano as fallback.

src/worker/jobs/ai-enrich-suggestion.job.ts
4

Approval Queue

Scheduler sees pending suggestions in the Queue page. Each card shows: student, proposed time, instructor, aircraft, AI rationale, risk score. Approve → sets status=processing (optimistic lock) → validates with FSP → creates reservation. Decline → marks declined. Bulk actions supported.

src/api/modules/suggestions/suggestions.service.ts
5

Auto-Expiration

Runs every 10 minutes. TTL-based: marks as expired when expiresAt < now. Slot-filled: checks if proposed slot was booked by someone else. Configurable TTL per operator (default 24h).

src/worker/jobs/expire-suggestions.job.ts
6

Notifications

Triggered on approval. Resolves email/SMS templates with variables ({{studentName}}, {{proposedDate}}, etc.). Dispatches via Resend (email) or Twilio (SMS). Records delivery status.

src/worker/jobs/send-notification.job.ts

⚙ Core Scheduling Engine

Pure TypeScript modules in src/core/ — no framework dependencies, fully testable.

slot-finder.ts — 416 lines
Slot Finding

Progressive window expansion (7→14→28 days). Filters by daylight hours (6:30am–6pm civil twilight), instructor/aircraft availability. Returns scored candidate slots.

constraint-evaluator.ts — 325 lines
Constraint Checking

Instructor currency + duty time. Aircraft maintenance. Student enrollment prerequisites. Location timezone conversions. Returns pass/fail with reasons.

waitlist-ranker.ts — 202 lines
Waitlist Ranking

Multi-criteria scoring with operator-configurable weights: time on list, curriculum progress, currency, total flight hours, priority level.

enrollment-analyzer.ts — 231 lines
Enrollment Analysis

Determines next required lesson in curriculum. Tracks progress %. Detects completions and pending lessons for next-lesson pipeline.

rationale-builder.ts — 142 lines
Rationale Building

Generates human-readable explanations: why this student, why this slot, what constraints were satisfied. Fed to AI for enrichment.

change-detector.ts — 173 lines
Change Detection

Deterministic hashing of schedule events. Compares to previous hash. Identifies specific cancellations and new gaps.

🔒 Security Architecture

Guard Pipeline

RateLimitGuard → AuthGuard → TenantGuard — applied globally in order. RateLimit: 100 req/min (10 for auth). Auth: JWT HS256, 8h expiry. Tenant: sets PG session var for RLS.

Encryption

AES-256 for sensitive data at rest (FSP tokens). JWT_SECRET (32+ chars). ENCRYPTION_KEY (64 hex chars). No plaintext secrets in DB.

HTTP Headers

X-Content-Type-Options: nosniff, X-Frame-Options: DENY, HSTS (1yr), XSS-Protection, Referrer-Policy, Permissions-Policy (no camera/mic/geo).

Multi-Tenant Isolation

Every table scoped by operatorId. TenantGuard sets PostgreSQL session variable app.current_tenant. Row-Level Security enforces at DB layer.

Every suggestion gets an async AI job. Inputs: student progress, instructor workload, constraints, historical patterns. Outputs: explanation summary, compliance list, risk level + reason.

Chat interface at /ask. Claude gets full database context (students, schedule, policies). Users ask natural language questions about their school's data.

Design principle: AI enhances, humans decide. AI generates rationale + risk scores, but the scheduler always approves or declines. AI enrichment is async — never blocks suggestion creation.

★ Key Design Choices

1. Event-Driven, Not Polling UI

BullMQ jobs detect changes → generate suggestions → enrich with AI. The scheduler just sees a ready queue. No manual slot-hunting needed.

2. Multi-Tenant Isolation

Every table has operatorId. TenantGuard + PostgreSQL RLS. One flight school never sees another's data. Policies configurable per tenant.

3. Progressive Search Window

Start 7 days → expand to 14 → then 28 max. Avoids overwhelming with distant options while ensuring coverage. Configurable per operator.

4. Optimistic Concurrency

status=processing lock prevents double-approval. If FSP validation fails, reverts to pending with error details stored in JSONB.

5. AI as Enhancement, Not Decision-Maker

AI generates rationale + risk scores, humans approve. Enrichment is async — never blocks suggestion creation. Fallback model if primary is down.

6. Mock Mode for Dev/Demo

FSP_MOCK_MODE=true replaces all 76 FSP endpoints with canned data. Idempotent seeder populates DB on startup. Full demo without real credentials.

7. Separation: API vs Worker

Same codebase, different entry points. API is stateless HTTP (Fastify for speed). Worker handles BullMQ jobs (no HTTP). Both in one Docker container in prod.

8. Pure Core Engine

src/core/ has zero framework dependencies. Slot finder, constraint evaluator, ranker — all pure TypeScript. Fully unit-testable in isolation.
